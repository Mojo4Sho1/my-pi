# HANDOFF_CONTRACT.md

## Purpose

This document defines the repository-level handoff system for `my-pi`.

The handoff system exists to maintain a clear, durable, low-ambiguity control plane for ongoing work across human-driven, Codex-driven, and orchestrator-driven execution. It defines:

- which handoff documents own which truths
- how the active task bundle is selected and advanced
- how progress, blockers, and decisions are recorded
- what must be updated before a loop can close
- how the repository-level handoff layer differs from downstream task packets

This contract governs the **repository handoff layer only**. It does **not** define the downstream packet format used by teams or specialists.

---

## Scope

This contract applies to the live handoff documents under `docs/handoff/`.

It governs:

- `NEXT_TASK.md`
- `CURRENT_STATUS.md`
- `TASK_QUEUE.md`
- `DECISION_LOG.md`
- `OPEN_DECISIONS.md`
- `_HANDOFF_INDEX.md`

This contract does **not** replace:

- agent definitions
- workflow/routing definitions
- downstream task packet contracts
- downstream result packet contracts

Those documents may refine behavior within their own scopes, but they must not contradict this contract.

---

## Core principles

### 1. Single active bundle
At any given time, the repository has **one active task bundle**.

A task bundle may contain multiple tightly related subtasks, but it must serve one coherent objective.

### 2. Repository bundle vs downstream packet
`NEXT_TASK.md` defines the **active repository-level task bundle**.

It is:

- human-readable
- repository-facing
- top-level-executor-facing
- orchestrator-facing

It is **not** itself a downstream specialist/team task packet.

Downstream task packets are separate execution artifacts generated by the orchestrator when delegated execution is used.

### 3. Minimal necessary context
The handoff layer must provide enough context for the next top-level executor to act safely and correctly, and no more.

It must:

- expose the active bundle clearly
- preserve key state and rationale
- avoid unnecessary historical noise
- avoid forcing downstream actors to read broad repo context by default

### 4. Forward motion under uncertainty
Unresolved decisions do not automatically stop progress.

The system should continue safely whenever it can do so without guessing in a way that creates likely rework or architectural drift.

### 5. Durable truth over implicit memory
If a state transition, durable decision, or active blocker matters for future execution, it must be written into the handoff layer.

The system must not rely on unstated memory.

---

## Execution contexts

The repository may operate in more than one top-level execution context.

### Direct bundle execution
A human or external executor such as Codex reads the repository handoff layer, executes the active bundle directly, and updates the handoff layer.

### Orchestrated execution
The orchestrator reads the repository handoff layer, interprets the active bundle, selects an execution strategy, generates downstream task packets as needed, integrates returned results, and updates the handoff layer.

### Contract rule
In both contexts, the handoff layer defines repository execution state.

Only the orchestrator may convert an active bundle into downstream task packets.

## Ownership of truth

Each handoff document has a specific job. Those jobs must remain distinct.

### `NEXT_TASK.md`
Owns the definition of the **single active repository task bundle**.

It answers:

- what bundle is active now
- why this bundle is next
- what is in scope
- what is out of scope
- what conditions define completion
- what conditions require escalation

`NEXT_TASK.md` is the sole execution selector for the repository.

### `CURRENT_STATUS.md`
Owns the current operational snapshot.

It answers:

- what the repository is currently focused on
- what has been completed in the current focus
- what important gaps or blockers remain
- what the current next target is at a high level

`CURRENT_STATUS.md` must remain concise and current.

### `TASK_QUEUE.md`
Owns the ordered planning pool.

It answers:

- which bundles are queued
- which bundles are deferred
- which bundle should become active after the current one closes
- how queued work relates to current state

`TASK_QUEUE.md` does not override `NEXT_TASK.md`. It supports it.

### `DECISION_LOG.md`
Owns durable decisions that have already been made.

It answers:

- what durable decision was made
- why it was made
- when it was made
- what future work must now conform to it

It does **not** serve as a generic progress log.

### `OPEN_DECISIONS.md`
Owns unresolved decisions that are not yet fully settled.

It answers:

- what decision remains open
- whether it is currently blocking
- what work may continue without it
- what condition would force escalation
- who owns eventual resolution

### `_HANDOFF_INDEX.md`
Owns discoverability within the handoff layer.

It answers:

- which handoff files exist
- what each handoff file is for
- where readers should start

---

## Priority and conflict resolution

If handoff documents appear to conflict, resolve them in this order:

1. `DECISION_LOG.md` for durable decisions already made
2. this contract
3. `NEXT_TASK.md` for the active bundle
4. `CURRENT_STATUS.md` for current snapshot
5. `TASK_QUEUE.md` for ordered future work
6. `OPEN_DECISIONS.md` for unresolved matters
7. `_HANDOFF_INDEX.md` for navigation only

If a conflict cannot be safely resolved, mark the affected work as `partial` or `blocked` and record the issue explicitly.

## Bundle lifecycle

A task bundle moves through explicit states.

### Allowed bundle states

- `queued`
- `active`
- `partial`
- `blocked`
- `complete`
- `deferred`

No other bundle state should be used unless a later durable decision explicitly introduces it.

### State definitions

#### `queued`
The bundle is planned but not currently active.

#### `active`
The bundle is the single active repository target.

#### `partial`
Meaningful progress was made, but the bundle did not fully meet its done criteria.

A `partial` outcome must record:

- what was completed
- what remains
- what caused the stop
- whether the remaining work is blocked or merely unfinished

#### `blocked`
No safe forward progress can currently occur on the bundle.

A `blocked` outcome must record:

- the blocker
- why the blocker prevents progress
- what change would unblock it
- whether escalation is required

#### `complete`
The bundle met its done criteria and its required handoff updates have been made.

A bundle is not complete until the handoff layer reflects completion.

#### `deferred`
The bundle is intentionally postponed in favor of other work.

The reason for deferral must be recorded.

## Active bundle rules

### Single active bundle rule
Only one repository task bundle may be `active` at a time.

### Bundle shape rule
A bundle may include several tightly related subtasks, but they must all support one coherent objective.

### Bundle closure rule
A loop is not closed until the active bundle has an explicit outcome and the required handoff documents have been updated.

### Bundle advancement rule
When the active bundle closes as `complete`, `partial`, `blocked`, or `deferred`, the handoff layer must either:

- promote a new bundle into `NEXT_TASK.md`, or
- explicitly state that no bundle is being activated yet and why

---

## Loop model

A loop is one top-level execution cycle against the active repository bundle.

### Loop start
A loop begins when the executor reads the handoff layer and accepts the active bundle defined in `NEXT_TASK.md`.

### Loop scope
A loop operates on one active bundle only.

### Loop end
A loop ends only after:

- the bundle outcome is determined
- required validations are recorded
- `CURRENT_STATUS.md` is updated
- `TASK_QUEUE.md` is updated
- `NEXT_TASK.md` is updated or intentionally left in place with clear state
- `DECISION_LOG.md` and/or `OPEN_DECISIONS.md` are updated if needed

### Loop continuity
If the bundle outcome is `partial`, the next loop may continue the same bundle or split off a new bundle, but the handoff state must make that transition explicit.

## Validation model

Validation levels are cumulative.

Each higher level includes the requirements of the levels below it.

### Level 1: `doc_check`
Confirms documentation-level correctness within task scope.

This includes:

- internal consistency of touched documents
- absence of obvious contradictions introduced by the change
- plausible references, headings, and stated relationships
- satisfaction of documentation-oriented done criteria

### Level 2: `local_consistency_check`
Includes all `doc_check` requirements, plus local repository consistency checks.

This includes:

- referenced files exist
- touched documents agree with one another locally
- read/write boundaries were respected
- handoff documents remain mutually consistent after update

### Level 3: `repo_validation`
Includes all `local_consistency_check` requirements, plus repository-level validation where appropriate and available.

This includes:

- relevant commands, checks, or validations were run when appropriate
- implementation-level behavior was verified where feasible
- concrete validation evidence was recorded clearly

### Validation declaration rule
Every active bundle must declare its required validation level.

### Validation completion rule
A bundle cannot be marked `complete` unless it satisfies its declared validation level.

## Decision handling

The handoff system distinguishes between durable decisions already made and unresolved decisions still open.

### Durable decisions
A durable decision belongs in `DECISION_LOG.md` when it:

- changes a governing rule
- freezes a structural choice
- resolves a recurring ambiguity
- establishes a constraint future work must obey

A durable decision entry must include:

- decision title
- decision statement
- rationale
- date
- downstream implications

### Open decisions
An unresolved decision belongs in `OPEN_DECISIONS.md` when it:

- has not yet been settled
- may matter later
- does not yet require immediate interruption of current work

An open decision entry should include:

- decision ID or title
- context
- why it matters
- current status
- whether it is blocking now
- work that can continue without it
- trigger for escalation
- owner

### Escalation rule
Escalate an open decision to Joe when at least one of the following becomes true:

- the decision blocks safe forward progress on the active bundle
- guessing would likely create avoidable rework
- the decision changes a durable architectural rule
- the decision exceeds delegated authority

### Non-blocking decision rule
If an unresolved decision does not block safe progress, record it in `OPEN_DECISIONS.md` and continue.

---

## Progress reporting rules

The handoff layer should preserve only the information needed for correct continuation.

### Record explicitly
Always record:

- active bundle identity and objective
- current outcome
- blocker details when relevant
- remaining delta for partial work
- validation level and validation performed
- durable decisions made
- unresolved decisions that may matter later

### Avoid unnecessary noise
Do not turn the handoff layer into:

- a verbose execution diary
- a duplicate of git history
- a duplicate of downstream result packets
- a catch-all note dump

If detailed execution evidence must be preserved, summarize it cleanly and point to its canonical location.

---

## Context discipline

The handoff layer should support disciplined context consumption.

### Top-level executor context
A top-level executor may read the handoff layer plus any explicitly required documents named by the active bundle.

### Orchestrator context
The orchestrator may read the handoff layer plus the broader routing and contract documents needed to interpret and delegate the active bundle.

### Downstream actor context
Downstream actors should not consume broad repository context by default.

They should consume orchestrator-issued task packets and only the materials explicitly named in those packets.

### Contract rule
Do not use `NEXT_TASK.md` as a substitute for a downstream task packet in delegated execution mode.

---

## Required handoff updates after bundle execution

When a loop ends, the executor must update the handoff layer in a way that preserves safe continuity.

### Always update
At minimum, update:

- `CURRENT_STATUS.md`
- `TASK_QUEUE.md`
- `NEXT_TASK.md`

### Update when needed
Update `DECISION_LOG.md` when a durable decision is made.

Update `OPEN_DECISIONS.md` when an unresolved decision is surfaced, changes state, becomes blocking, or is resolved.

Update `_HANDOFF_INDEX.md` only when the handoff file set or navigation needs change.

### Completion rule
A bundle is not fully closed until these updates are done.

---

## Rules for `NEXT_TASK.md`

This contract does not define the full schema of `NEXT_TASK.md`; that belongs in `NEXT_TASK_CONTRACT.md`.

However, `NEXT_TASK.md` must always:

- identify the active bundle clearly
- define the bundle objective
- define scope in and scope out
- define done criteria
- define escalation conditions
- declare the required validation level
- remain repository-level rather than downstream-packet-level

If execution is delegated, `NEXT_TASK.md` informs packet generation but does not replace it.

---

## Rules for `TASK_QUEUE.md`

`TASK_QUEUE.md` must remain consistent with the active bundle and current state.

It should:

- show the active bundle’s relationship to the queue
- identify likely next bundles clearly
- preserve deferred work visibly
- avoid becoming an unstructured backlog dump

The queue exists to support correct updating of `NEXT_TASK.md`.

It does not independently authorize execution.

---

## Rules for `CURRENT_STATUS.md`

`CURRENT_STATUS.md` must remain a concise operational snapshot.

It should state:

- the current focus
- recently completed progress that matters to continuation
- key blockers or open gaps
- the current next target at a high level

It should not become a long historical archive.

---

## Rules for `DECISION_LOG.md`

`DECISION_LOG.md` must remain high-signal.

It should log decisions, not routine progress.

Do not log ordinary completions, minor edits, or non-durable observations unless they materially establish a new rule future work must obey.

---

## Rules for `OPEN_DECISIONS.md`

`OPEN_DECISIONS.md` must remain actionable.

It should surface unresolved decisions in a way that helps future work continue safely.

It must distinguish between:

- non-blocking decisions being monitored
- pending decisions likely to matter soon
- currently blocking decisions
- resolved decisions ready to archive or remove

---

## Safe behavior under ambiguity

When ambiguity appears, prefer the following order:

1. check durable decisions already recorded
2. check this contract
3. check the active bundle definition
4. continue safely if the ambiguity is non-blocking
5. record the ambiguity as an open decision if it may matter later
6. escalate only if the ambiguity blocks safe progress or exceeds authority

Do not silently invent policy.

Do not broaden scope to “clean things up” unless the active bundle explicitly permits it.

---

## Definition of handoff health

The handoff layer is healthy when:

- one active bundle is always clear
- current state and queue are mutually consistent
- bundle outcomes are explicit
- blockers are explicit
- durable decisions are logged
- open decisions are visible without causing unnecessary interruption
- top-level executors can resume work without guessing
- downstream actors receive only the context they need through proper packets

---

## Amendment rule

This contract is durable but amendable.

Amend it when a durable decision changes handoff semantics or when system maturity requires a clearer or broader control-plane model.

Any amendment to this contract must be treated as a durable decision and logged in `DECISION_LOG.md`.

---

## Summary rule

Use the handoff layer to define repository execution state.

Use `NEXT_TASK.md` to define the single active repository bundle.

Use downstream task packets to define delegated worker execution.

Record durable decisions explicitly.

Record unresolved decisions without interrupting safe progress unless they become true blockers.
